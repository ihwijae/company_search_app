# 공용폴더 수정사항 정리

## 배경

기존에는 사용자가 업로드 화면에서 전기/통신/소방 엑셀 파일을 직접 선택해 경로를 저장하는 방식이었다.

이후 공용폴더 기준 자동 탐색 방식으로 변경되면서:

- 구글드라이브 데스크탑의 `...\내 드라이브\협정보조툴` 또는 `...\My Drive\협정보조툴` 아래에서
- 파일명에 `전기`, `통신`, `소방` 키워드가 포함된 최신 xlsx 파일을 자동 탐색하고
- 해당 파일을 실제 검색 데이터 소스로 로드하는 구조가 되었다.

## 실제로 발생한 문제

`지역사찾기`에서 결과가 없다고 나왔지만, 일반 검색 쪽은 정상 경로를 보고 있었다.

로그 확인 결과:

- 일반 검색/자동탐색은 `loadedPath = F:\내 드라이브\협정보조툴\...xlsx` 를 사용
- 하지만 `지역사찾기` 후보조회 로직은 예전 설정값인 `configuredPath = F:\내 드라이브\...xlsx` 를 fallback 으로 참조할 수 있는 구조였다
- 이 경로에는 `협정보조툴` 폴더가 빠져 있었음

즉 공용폴더 전환 후 실제 사용 경로와 설정에 남아 있는 예전 경로가 달라졌고, `지역사찾기`만 이 차이의 영향을 받았다.

## 원인

핵심 원인은 검색 관련 기능이 모두 동일한 경로/동일한 로드 서비스만 보지 않았던 점이다.

특히 `agreements-fetch-candidates`(지역사찾기 후보조회) 쪽은:

- `SearchService`가 이미 로드한 데이터를 직접 참조하지 못하는 경우가 있었고
- fallback 시 `FILE_PATHS[fileType]`를 다시 읽었다
- 자동모드에서 `FILE_PATHS`에는 예전 수동 경로나 오래된 설정값이 남아 있을 수 있었다

결과적으로:

- 일반 검색은 공용폴더 자동탐색 결과 사용
- 지역사찾기는 예전 설정 경로 fallback 가능

이라는 경로 불일치가 발생했다.

## 이번에 적용한 수정

### 1. 지역사찾기도 동일한 SearchService 인스턴스를 사용하도록 수정

`main.js`

- `SearchService` 인스턴스를 모듈 스코프의 `searchServiceInstance`로 보관
- `agreements-fetch-candidates`에서 이 인스턴스를 직접 참조하도록 변경
- 이제 지역사찾기 후보조회도 일반 검색과 동일한 로드 데이터 소스를 우선 사용함

### 2. fallback 경로도 자동모드 기준으로 다시 계산하도록 수정

`main.js`

- 지역사찾기 fallback 시 `FILE_PATHS[fileType]`만 보지 않도록 수정
- 자동모드일 경우 `resolveAutoDiscoveredExcelPath(fileType)` 또는 `resolveLatestMatchingExcelPath(...)`를 다시 사용
- 즉 fallback 이 필요해도 공용폴더 자동탐색 결과를 우선 보도록 변경

### 3. 자동모드에서 실제 사용 경로를 config 에도 동기화하도록 수정

`main.js`

- `syncResolvedFilePath(fileType, resolvedPath)` 추가
- 자동모드에서 실제 로드된 경로(`loadedPath`)가 기존 `FILE_PATHS[fileType]`와 다르면:
  - `FILE_PATHS`를 갱신
  - `saveConfig()` 호출
- 앱 시작 시 preload 로드 직후에도 동기화
- 파일 변경 감지 후 재로드될 때도 `notifyUpdated` 경로에서 다시 동기화

의도:

- 자동모드의 실제 사용 경로를 `config.filePaths`에도 반영
- 다른 기능이 `FILE_PATHS`를 직접 보더라도 예전 잘못된 경로를 참조하지 않도록 완화

## 현재 기준 동작

### 수동 모드

- 사용자가 직접 선택한 경로를 `FILE_PATHS`와 `config`에 저장
- 자동 동기화 로직이 수동 경로를 덮어쓰지 않음

### 자동 모드

- 공용폴더 자동탐색으로 실제 로드한 경로를 사용
- 그 실제 경로가 기존 설정값과 다르면 `config.filePaths`도 갱신
- 즉 자동모드에서는 `config`가 “실제 사용 경로의 동기화된 값” 역할을 함

## 확인했던 진단 로그

문제 확인용으로 `agreements-fetch-candidates`에 아래 로그를 추가했다.

- `fileType`
- `sourceMode`
- `configuredPath`
- `loadedPath`
- `fallbackPath`
- `serviceSearchAttempted`
- `serviceSearchReady`
- `serviceSearchFailed`
- `dataSource`
- `rawCount`
- `excludedAlwaysCount`
- `excludedSingleCount`
- `excludedRegionCount`
- `finalCount`

문제 당시 확인된 핵심 로그:

- `configuredPath`: `F:\내 드라이브\2.협력업체요약...xlsx`
- `loadedPath`: `F:\내 드라이브\협정보조툴\2.협력업체요약...xlsx`

이 차이로 인해 원인이 경로 불일치임을 확인했다.

수정 후 정상 로그 예시:

- `serviceSearchAttempted: true`
- `serviceSearchReady: true`
- `dataSource: 'searchService'`
- `loadedPath`가 `...\협정보조툴\...` 로 표시됨

## 성능 영향

이번 수정으로 성능 저하 가능성은 거의 없다.

- 원래도 자동탐색과 엑셀 로드는 수행 중이었음
- 추가된 것은 경로 문자열 비교와 작은 `config.json` 저장 정도임
- 저장도 “자동모드 + 실제 로드 경로가 기존 설정과 다를 때만” 발생

병목이 생길 가능성이 있다면 엑셀 재로드가 원인이지, config 동기화 자체는 아님

## 다음 세션 참고사항

추가로 점검하면 좋은 항목:

- 전기/통신/소방 엑셀을 직접 다시 읽는 다른 기능이 남아 있는지 확인
- 검색/지역사찾기 외에도 `FILE_PATHS`를 직접 참조하는 코드가 있는지 점검
- 가능하면 검색 관련 기능은 모두 `SearchService` 또는 그 로드 결과만 보도록 정리
- 진단 로그는 문제 재발 방지를 위해 한동안 유지해도 됨
